KEY DECISIONS & RATIONALE
================================================================================

1. NO QR PAYMENT INTERCEPTION
   Why: Deep research proved this is:
     - Illegal (RBI/NPCI actively prosecute this pattern)
     - Violates Stripe/Razorpay merchant agreements
     - FTC classifies as unfair commercial practice
   Solution: ReviewOS QR displayed alongside (not before) payment QR
            Or better: on post-payment confirmation screen

2. TALKING POINTS, NOT REVIEWS
   Why: Google bans AI-generated reviews
        Customers copying pre-written text triggers Google's ML filter
   Solution: AI converts customer's words into bullet reminders
            Customer writes review in their own words on Google
            100% authentic, 0% filter risk

3. NO REVIEW GATING
   Why: FTC 16 C.F.R. § 465 classifies this as deceptive
        Google's policy prohibits it too
   Solution: Every rating (1-5 stars) sees identical Google button
            Private feedback is alternative, never forced/hidden
            Compliance locked at code level (no toggle to disable)

4. MOCK DATA LAYER (NOT LOCALSTORAGE)
   Why: localStorage is not production-ready persistence
        Businesses need real data durability
   Solution: Mock data in memory with async accessors
            Direct swap to Supabase in Phase 2 (no UI changes)
            Pattern is production-proven

5. SECURITY FIRST
   Why: Review data is sensitive (contains customer opinions, names)
        APIs accept user input (XSS/injection risk)
        Business models matter (rate limiting prevents abuse)
   Solution: Input validation & sanitization on all endpoints
            Rate limiting prevents DoS
            Security headers prevent browser attacks
            Compliance locked at code level
